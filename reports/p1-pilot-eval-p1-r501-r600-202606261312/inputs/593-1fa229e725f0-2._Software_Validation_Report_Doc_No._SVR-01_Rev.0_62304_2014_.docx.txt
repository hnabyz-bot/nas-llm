Definitions and Terms

The definitions and terms used in this specification are as follows.

▪ Harm

physical injury or damage to the health of people, or damage to property or the environment

▪ Hazard

potential source of harm

▪ Hazardous situation

circumstance in which people, property, or the environment are exposed to one or more hazard(s)

▪ Intended use/ intended purpose

use for which a product, process or service is intended according to the specifications, instructions and information provided by the manufacturer

▪ Life-cycle

all phases in the life of a medical device, from the initial conception to final

decommissioning and disposal

▪ Post-production

part of the life-cycle of the product after the design has been completed and the medical device has been manufactured

▪ Residual risk

risk remaining after risk control measures have been taken

▪ Risk

combination of the probability of occurrence of harm and the severity of that harm

▪ Risk analysis

systematic use of available information to identify hazards and to estimate the risk

▪ Risk assessment

overall process comprising a risk analysis and a risk evaluation

▪ Risk control

process in which decisions are made and measures implemented by which risks are

reduced to, or maintained within, specified levels

▪ Risk estimation

process used to assign values to the probability of occurrence of harm and the severity of that harm

▪ Risk evaluation

process of comparing the estimated risk against given risk criteria to determine the

acceptability of the risk

▪ Risk management

systematic application of management policies, procedures and practices to the tasks of analyzing, evaluating, controlling and monitoring risk

▪ Safety

freedom from unacceptable risk

▪ Severity

measure of the possible consequences of a hazard

Architecture Design Chart

Architecture Design

Initialize default data Module

The initialize default data module manages the default data. It can initialize the default data of microprocessor, interrupt and timer. When the system starts, this module initialize the data of microprocessor, interrupt and timer to default data which is saved in the memory.

Operate module

The operate module manage the all the mode of the MINUET LCD & MINUET LCD eco. It adjust the suction pressure by controlling the RPM of the motor and the solenoid depending on the mode. When the user inputs items such as mode, step and pattern through the LCD, It adjust the output.

Battery level scan module

The Editing Module check the battery level. When the module completes the battery level check, it sends the information to the LCD user interface module so that the user can confirm it

Vacuum pump module

The Vacuum pump module is responsible for controlling the RPM of the motor. It inputs the operation information from the operate module.

Solenoid valve module

The Vacuum pump module is responsible for controlling the operation of solenoid. It inputs the operation information from the operate module.

LCD User interface module

The LCD user interface module is responsible for indicating the data. When information is received from another module, it is displayed on the LCD according to the information. It also controls the information entered by the user via the button.

Error module

If unexpected error is occurred, the device will set the watch dog flag it will reset the program.

Safety Vmodule

The safety module is responsible for stopping the device. It stop the operation of the MINUET LCD & MINUET LCD eco after 30minutes of use.

Software verification & test plan

Overview and Purpose

Ensure compliance with the user requirements, the developed software during the software lifecycle of our Software and check whether it Contradict with the detailed design information, and check it works properly when integrate each unit as plan. Verify the validity in the process of each test and verification of control measures as a means of risk control.

Software integration and integration testing Planning

Once the software design is completed, TEST the interaction between the developed program and other system (Operating System) after checking that the unit program or module functions peorplery as defined in the program specification. Reference: 2.1 Software development process

Technique and test procedure for the test method

Testing Methods

Static Testing & Dynamic Testing

Testing Description

(1) Static Testing Outline

Static Testing is a method to find defects without executing the software. The code will be reviewed by the programmer who wrote the code and other programmers to find the problems and things that are missing, and record its result

(2) Level of testing

The test is processed in a step-by-step process (from the sub-code to the module), and integration and integration verification is performed.

(3) Dynamic Testing Outline

Dynamic Testing is a method to find defects by executing the software. Found defect is modified by debugging, and is a representative method of a direct test.

(4) Functional or Black box testing

Functional testing is a technique in which functionality and requirements of the software are tested without looking at the internal design structure.

It proceeds by performing a dynamic testing focused to the requirements and functionality.

Clearly state the test criteria and the requirements of all clients to minimize errors that may occur in actual operation.

Solution process when defect occurs

Test process follows the following order.

Error found

If an error is found while performing the requirement analysis, design, and test process, please discuss with the professional testers and project teams.

Error registration

Register the error in the defect management report.

Error analysis

Analyze whether the registered error is just the simple error or the physical defect.

Defect confirmation

When the registered information is confirmed as a ‘defect’, set the found error to ‘defect state’

Defect assignment

Specifies the representative and assign the defect.

Defect action

Perform a corrective action of the defect, and if corrective action is completed,

set the defect as the ‘defect action state’

Defect action review and approval

Perform verification testing of the modified defect and ensure that the defect is corrected properly. If the action is completed successfully, set the defect as ‘action complete’ state.
