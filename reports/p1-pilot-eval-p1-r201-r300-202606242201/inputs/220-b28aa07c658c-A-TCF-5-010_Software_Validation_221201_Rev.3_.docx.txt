3.Software Requirements Specification (SRS)

3.1 General Specification

3.1.1 Terms and abbreviations

Frame – whole data frame received from detector ”as is” with original width and height.

ROI – abbreviation for “Region Of Interest“ (ROI assumed to be located within a frame in this document’s context)

Image – given ROI with optional rotation and flip. In particular cases it might coincide with the whole frame.

End User Application (EUA) – upper level software that acquires images from detector using this library and then presents (displays) images to end-user.

UI – abbreviation for User Interface

SU – SetUp. It is Property sheet supplied by Software for intrinsic configuration and maintenance operations.

Offset/Gain-calibration –equalization of pixels’ transfer functions

Bad pixel – pixel that can’t be calibrated

CBP – CallBack Process. Acquisition SW(DLL) passes data to Upper level SW(EAU) through CBP(data : a error code and a acquisition frame).

3.1.3 Role of Software

This is “middle-ware” utility library between still image detector (and its low-level driver) from one side and EUA from the other. It also contains SU to modify software and hardware configuration parameters and preparation of Gain/Offset-Calibration and Bad Pixels’ Correction. Software shall be able also to perform some image processing operations, which are stipulated in SRS.

Images from Software shall be generally ready for display. But fine final enhancement of visual presentation is not an exact function of this Software. A customer (physician) might require further post-processing (like automatic or manual contrast and brightness adjustment or some additional filtration algorithms. EUA should be responsible for such post-processing.

3.1.6 Data Formats

All image data shall be transmitted and stored as 16-bits little-endian (Intel bytes order) two- dimensional arrays without any specific header. Image’s dimension shall be passed as auxiliary procedure’s parameter or value in configuration file. Grayscale range assumed to be 0..1^16-1 (16 bits non-negative integer) for saved files. That is eligible if Software internally uses either signed and unsigned integer types for pixels’ values.

Raw frame’s pixels are 14 bits non-negative integers. Software treats highest unused bits of frame’s pixels as zeros (even detector’s hardware set those to “1”).

4.Software Design Specification (SDS)

4.1 Introduction

Purpose of this document is to describe relations between SRS, source code and User Manual (refer paragraph 2.2 for prerequisites). This document does not duplicate comprehensive and detailed consideration of Software architecture in the User Manual. This document enumerates (enlists, formalizes) items of SRS and User Manual for tracebility analysis. The expected audience of this document would be Software Engineer, who needs to modify functionality of this Software.

4.2 System Overview

Refer also paragraph 2.1 in SRS.

This Software is a part of workspace of Still Imaging X-ray Acquisition Library. The workspace contains several acquisition libraries’ projects. The projects share most of their source code, that implements general operations for still image X-ray detectors. All projects use the same header files at the highest abstraction level. All projects have almost identical SW-interface (of course meaning, range and scope of control parameters are specific for each detector’s type).

4.3 Design Considerations

4.3.1 General Constraints

Software invokes only commnon subset of Windows API communicating with Ethernet detector to conform Srs-C-DepE. Source code follows the syntax of MS VC 2015 and contains respective pragmas under conditional “define” (file stdafx.h) for more recent version of C++ compiler from MS Visual Studio 2019.

To make deployment easier, Software never relies on existence of MFC (as well as all of another specific libraries) in system’s paths but has those linked statically.

Still imaging detectors do not produce huge data-series and do not require especially high performance during acquisition. So we need not to impose especial constraints for available physical memory or CPU frequency.

4.3.2 Goals and Guidelines

Purpose of this library’s architecture is to provide optimal trade-off between abstraction level and customization of particular X-ray detector libraries. A DLL simply mapped into address space of caller process gives most flexible and customizable application interface.

Note that this architecture is not suitable for fast acquisition. For fluoroscopy or CT we would rater prefer to run acquisition/reconstruction as separate process in (executable program) its own address space to isolate EUA from potential danger of exceptions in acquisition modules and resources’ leakage.

5.Hazards’ Analysis

Introduction

This document contains results of the Hazard Evaluation and corrective actions for those hazards for the Software. The model used for the Hazard Evaluation was Failure Mode and Effects Analysis (FMEA). Used terms are defined below:

The probability concerns hazards that occur during normal system operation.

High (H): Likely to occur more than 30 times per installation site and year. Medium (M): Probability between High and Low.

Low (L): Likely to occur 2 or less times a year.

Eliminate (E): Probability shall be reduced near to zero. “Zero” means that problem virtually shall happen rarer than “blue screen of death” because of failure of basic hardware like hard disk, video-adapter or mainboard).

The level of concern considers the difficulty to detect and handle the origin of a hazard and also the possible harm that may come to a patient. Possible harm to a person takes precedence over the difficulty to detect and handle the origin of the hazard.

Minor: The hazardous condition does not result in injury to the patient.

Moderate: The hazardous condition could directly cause minor to moderate injury to the patient, or indirectly cause minor to moderate injury via incorrect or delayed information being presented to

the care provider.

Major: The hazardous condition could directly cause death or serious injury to the patient, or indirectly cause death or serious injury via incorrect or delayed information being presented to the care provider
