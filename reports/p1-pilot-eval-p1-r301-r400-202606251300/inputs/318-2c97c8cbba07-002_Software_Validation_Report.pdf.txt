✓ xSystem Module : System initialization and task creation

2.1. Software Name : SDK-MCW
2.1.1. Software Function
✓ Common functions module
The module communicates with the detector through a network socket communication module.
Common functions module provides commands to control detector hardware such as system
initialization, window time setting for image acquisition and detector operation mode setting, and
exposure time setting. It also receives feedback on the condition of the hardware from the
detector.
✓ Image acquisition module
The Image Acquisition module registers the event handle setting for acquisition of image
capture images, data processing event settings, and notification processing messages for

image acquisition, and stores the raw images read from the detector.
✓ Calibration module
For proper image correction, information of the beam delivery efficiency of each fiber bundle is
very important. Therefore, information of the beam delivery efficiency of each fiber will be
provided in the calibration dataset. The calibration dataset will produce a table of deviation of
fiber bundle beam delivery efficiency, and apply it to acquired images for flat fielding of the
image.
✓ Firmware update module
The Firmware update module provides a function to read the firmware information of the
detector hardware or write the latest firmware, and provides commands to manage the firmware
of the detector hardware.
✓ xSystem module
When communicating with the SDK, the xSystem module creates each task that manages and
controls the detector device, and supports reading and writing log data related to the shock
sensor.
✓ FPGA App module
The FPGA App module sends and receives messages when communicating between the CPU
and the FPGA, and supports functions to transmit image acquisition commands to the FPGA,
set the operation mode, and store and process the transmitted image data.
✓ System task module
System task module supports and monitors functions such as Wi-Fi operation status
management, power cable, power button, FPGA F/W erase and write function, and network
status.
✓ Interface module
Interface module supports i2c expansion controller control for peripheral devices, power supply
communication control, and shock sensor-related chip initialization and register read/write

4. Application Algorithm and Technology
Below are algorithm, mathematical expression and technology used in SDK-MCW.
4.1. Calibration
Calibration performs following compensations to acquired images.
4.1.1. Geometry Correction
In order to compensate geometric error caused by bidirectional scan, it performs bidirectional
Sync.
Because of characteristics of swing mirror driving, flying spot position of laser is determined by
tangent and sine function and angle speed. And the angle speed is calculated by time taken
between 2 sensors positioned at each end side of imaging area. Through this compensation,
acquired pixels are coordinated.
(inverse function with scale factor b)
To prevent deterioration in resolution caused by discordance between acquired pixel
coordination and compensated pixel coordination, extract 3 times of pixel number in horizontal
direction than required pixel numbers, then calculate appropriate sub pixel intensity for real
coordinate.
4.1.2. Offset Correction
Obtain dark image which noise is filtered. And then, subtract each pixel value of dark image
from real X-ray acquired image. This correction process makes dark signal offset uniform, and
enables to get uniform image.
4.1.3. Gain Correction
Gain map is calculated through calibration steps described below.
Acquire images which intensity levels are 2%, 20%, 40% and 80% of full dynamic range and
remove noise and artifact. Calculate ratio of optical fiber intensity level and standard intensity
level. Considering image characteristics of x-ray storage phosphor plate, acquire intensity
information on 3,500 pixels in horizontal direction and 128 pixels average Kernel intensity
information in vertical direction. Through this step, 2D gain map is generated. This 2D gain
map is applied to real scanned image by multiplying gain to each corresponding pixel.
4.1.4. Defect Correction
In case that the scanner can’t detect normal intensity light because of alien object in optical fiber
collection part or X-ray storage phosphor plate, extract corresponding area and compensate
through dust remover and defect correction algorithm.

✓ Common functions module
The common functions module communicates with the detector through the network socket
communications module. The common functions module provides commands for controlling
detector hardware, including system initialization, window time settings for image acquisition,
detector operational mode settings, and exposure time settings. Also provide hardware status
feedback with the detector.
✓ Image acquisition module
The Image acquisition module registers event handle setting and data processing event setting
for acquiring a video image, and notification processing messages about image acquisition, and
stores the raw image read from the detector.

For proper image correction, information of the beam delivery efficiency of each fiber bundle is
very important. Therefore, information of the beam delivery efficiency of each fiber will be
provided in the calibration dataset. The calibration dataset will produce a table of deviation of
fiber bundle beam delivery efficiency, and apply it to acquired images for flat fielding of the
image.

The image processing module performs image processing on raw images. Contrast the
background level data and images contained in the Raw image, adjust the noise reduction level
and contrast between light and shade, and remove the defect pixel contained in the Raw image.
It also applies the settings of the specified parameters and processes them to apply the optimal
values when deleting the data of the noise components contained in the raw image from the
object image.
✓ Firmware update module
The Firmware update module provides the ability to load detector hardware firmware
information, use the latest firmware, and provide commands to manage detector hardware
firmware.
✓ xSystem module

communication with the SDK and to read and write log data associated with impact sensors.

and supports the ability to transfer image acquisition commands and set operating modes to the

7. Software Development Environment Description
7.1. Introduction
This procedure manual (PM) describes the procedure for the development process. It will
allow medical software to meet all legal requirements when being developed. Development
faults will be avoided by means of a planned development process with an integral risk
management system. A description is given on how the necessary testing, verification and
validation steps are to be taken, from the Software Requirement Specification through the
release of the software and how changes to the software are handled. There are also
guidelines for incorporating amendments. The development process fulfils the requirements
21 CFR 820.30 Design Controls.
7.2. Concepts
Verification (suitability for the application)
✓ Verification confirms that the objectively defined requirements have been met.
Validation (suitability for implementation)
✓ This confirms that software has been tested for usability against all items in the
requirements specification. It must also satisfy all the internal requirements and
demands of the user.
PEMS : Programmable electrical medical system
PESS : Programmable electronic sub-system

7.3. Software Life Cycle
Software development process at SDK-MCW follows a classic waterfall methodology, as
illustrated in the following chart

Analysis. The activity consists of establishing requirements for as much as possible before the design
phase. The requirements are documented in the software-specific requirements and design notes. When
enough of the requirements are gathered and analyzed for this iteration of the prototype, the activities shift
into the design phase.
Planning. This activity constructs the software development plan. This plan outlines the tasks,
responsibilities, resources, and other items pertinent to the specific development project.
Design. This activity considers four functional attributes of the software: data structure, software
architecture, functionality, and interface characterization. The design is documented in design documents.
Implementation. The implementation phase converts the design into a machine readable form. The
software source code is created in accordance with SDK-MCW's coding conventions. To verify that the
code created is consistent with Company standards, design reviews/inspections are held.
Qualification. The qualification phase verifies that the design and code implemented meet the
requirements. Testing ensures that defined input will produce actual results that agree with required results.
Maintenance. Once the software enters a released state, the only software changes allowed are to fix
identified defects in the code or to support approved enhancements. All identified system software defects
are recorded on System Discrepancy Reports (SDR) forms or submitted electronically to the Modification
Request (MR) database.

7.4. Module Test (Verification)
In principle, each requirement from the specifications as well as from risk analysis must be
verified by means of an appropriate test procedure.
The following methods are used:
✓ Testing at component and/or system level
✓ Code inspection (for untestable requirements)
✓ Reviewing, examination of the documentation
7.4.1. Verification of Components
The development of software components takes place in a cyclical process of implementing and
testing of components. The test is performed by the programmer involved. The basis is the
requirements assigned to the components and any test methods and acceptance criteria
additionally specified in the design document.
When all requirements have been met, the components are authorized and/or made available
for use in the integration
It is specified for each component in the design document, whether it is to be subjected to a
component test or whether it is to be verified in the system test in interaction with other
components. When a component test is performed, test documentation must be created.
This is the project leader’s responsibility.
7.4.2. System Verification
The verification of the final product is done by means of simulated client. The fulfilling of
system requirements is to be verified by means of one or more test procedures.
Documentation must be created.
The proof of a subset of requirements can be done in component tests. In such a case the test
results for each component in question must be submitted.
The authorization / rejection of the software takes place after the test results have been
submitted in the verification review. The development leader, head of sales and quality
assurance official or a representative must take part in this. It is tested whether the software
coincides with the demands and whether the validation phase can be proceeded with.
The protocol for the verification is an informal document (minimum contents: participants, date,
software version involved, result of the discussions, new tasks with deadlines and
responsibilities, software version).
7.4.3. Test Documentation
The documentation of a test must be done in such a way that the test is made comprehensible
and repeatable.
A test procedure is drawn up in preparation of the test. The document must contain a
description of how the test has been drawn up and of the test data, as well as an overview of
the test procedure. In addition, the test steps to be performed and their expected results are
listed in tabular form.

7.5. System Test (Validation)
Requirements and testing for validation are laid down as for verification, whereby the validation
tests are by definition restricted to the use of a possible end user.
Participants in validation are: developments leader, head of sales, quality assurance official.
For validating the end product a simulated end user test, whereby the intended use is imitated
by using simulation software and corresponding hardware and following the instruction manual
and service documentation.
Then the conformity assessment is conducted. The validation of the product is repeated with a
real user test (by clients) with the provision that the product has not yet been certified. By the
systematic application of the medical product observation and reporting system, the result of the
validation is continuously reviewed and hardened up.
Particularly relevant observations regarding the product’s behaviour under working conditions
(e.g. by evaluating the service reports etc.) are regularly evaluated by the development head in
collaboration with the security official for medical products. In addition, the risk analysis is
assessed and expanded in each case if needed.
7.6. Amendments
After completion of the development and authorisation of the zero series, the product may
basically only be changed or modified by a development change. Changes are hardening the
source code as well as new minor and lead to a service pack. Debugging is corrective
measure for removing faults and is not defined as a development change.
Development changes only occur when the client’s product requirements or marginal technical
conditions change. The requirement specifications for changes are laid down in the minutes of
the developer meetings. There, all requirements are specified and tested at the same time for
any possible new risks. If so, the risk analysis is adjusted.
The amendments must be documented and are added to the main product file.
✓ new conformity statement (see manual)
✓ new manual
✓ filing the program CD
✓ change files with description of changes to software
Any amendment is concluded when a new service pack is released.

13. Unsolved Anomalies
13.1. Scope
List of errors and bugs which remain in the device and an explanation how they were
determined to not impact safety or effectiveness, including operator usage and human factors.
For each anomaly, indicate the problem, the impact on device performance, and if appropriate,
any plans or timeframes for correcting the problem. This list of bugs should be communicated
to the user in the device labeling.
✓ Safety
Patient, and/or user safety issue. Product cannot be released without compromising one
of these. Must be fixed.
✓ Major
Defect impacting product intended use. No work around available.
✓ Moderate
Impact on Product, typically related to user interface or general quality of product. Work
around is available
✓ Minor
No impact on product intended use, not significant or impacting product release, cosmetic.
