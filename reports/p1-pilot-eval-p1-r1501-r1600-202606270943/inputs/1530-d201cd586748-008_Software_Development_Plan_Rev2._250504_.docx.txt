Purpose of the software development plan

To use the Portable X-ray system(HnX-P1, HnX-PB) developed, produced and sold by this company according to intention, it aims at developing software that can manipulate and utilize the product function.

(HnX_P1 and HnX_PB, the software for HnX-P1 and HnX-PB, are the same software with different names.)

Work that should be performed in the development stage

(1) Determine which product will be developed, and forward the requirements to the R&D team so get it developed according to intention.

(2) Stage of the development work (time frame)

1) First stage : Select the product to be developed.

2) Second stage : Gather material on the selected product.

3) Third stage : Based on the gathered material, select and contract with a development firm which will realize the product development.

4) Fourth stage : Forward the requirements necessary for the product development to the selected firm.

5) Fifth stage : Once the development firm get started the product development, this company shall forward the development direction, changed requirements, etc to the development firm aside from a relevant meeting and support the product development positively.

6) Sixth stage : If the development is finished, conduct testing and verification for the product realization jointly with the development firm, and adjust, complement so make the completed product a reality.

Classification of software and its application

Software classification comply with FDA Guide Line

S/W is classified and applied in accordance with the table as follows. :

 “Major” Level of Concern

The level of concern is “Major” if a failure or latent flaw could directly result in death or serious injury to the patient or operator. The level of concern is also “Major” if a failure or latent flaw could indirectly result in death or serious injury of the patient or

operator through incorrect or delayed information.

 “Moderate” Level of Concern

The level of concern is “Moderate” if a failure or latent design flaw could directly result in minor injury to the patient or operator. The level of concern is also “Moderate” if a failure or latent flaw could indirectly result in minor injury to the patient or operator through incorrect or delayed information.

 “Minor” Level of Concern

The level of concern is “Minor” if failures or latent design flaws are unlikely to cause any injury to the patient or operator.

Software classification in accordance with EN 62304

S/W is classified and applied in accordance with as follows. :

This company shall assign software safety class to each software system originated by our software system and based on the impact of the harm on the patient, the worker, or others.

The software safety classes shall initially be assigned based on concern as follows:

-. Class A : There is no injury nor health damage.

-. Class B : The possible harm is non-serious injury.

-. Class C : The possible harm is death or serious injury.

In case harm is caused by that Software system is not operate by designated, the possibility of such defectiveness should be regarded as 100%.

If the risk of death or serious injury arising from a software failure is subsequently reduced to an acceptable level (as defined by ISO 14971) by a hardware risk control measure, either by reducing the consequences of the failure or by reducing the probability of death or serious injury arising from that failure, the software safety classification may be reduced from C to B. and If the risk of non- serious injury arising from a software failure is similarly reduced to an acceptable level by a hardware risk control measure, the software safety classification may be reduced from B to A. The our company assigned to each software system that contributes to the implementation of a risk control measure a software safety class based on the possible effects of the hazard that the risk control measure is controlling.

[If applicable] The company documented the software safety class assigned to each software system in the risk management file.

[If applicable] When a software system is decomposed into software items, and when a software item is decomposed into further software items, such software items inherit the software safety classification of the original software item (or software system) unless the our company documents a rationale for classification into a different software safety class. Such A rationale explains how the new software items are segregated so that they may be classified separately.

[If applicable] The our company documented the software safety class of each software item if that class is different from the class of the software item from which it was created by decomposition.

[If applicable] For compliance with standard, wherever a process is required for software items of a specific classification and the process is necessarily applied to a group of software items, the our company used the processes and tasks which are required by the classification of the highest-classified software item in the group unless the our company documents in the risk management file a rationale for using a lower classification.

[If applicable] For each software system, until a software safety class is assigned, Class C requirements applied.

Software Architectural Design

The software architectural design is a structure that explains the software requirements by the software structure and identifies the software item. The software architectural design includes a similar description etc on the relation between the primary functions of the software inclusive of the relation with the data flow such as a typical flow chart, hardware and networking. That does not need to include every function call of this document and module in general. However, there should be sufficient information to allow review by an organization on the functionality and intended use of the software equipment.

Software Design Specification

The software design specification explains materialization of the requirements on the software device. In the aspect of the relation between the Software Requirements Specification (SRS) and the Software Design Specification (SDS), SRS explains how the software device will be worked while SDS explains how the requirements will be materialized to SRS. The information presented to SDS should be enough to confirm the job conducted by a software engineer who made the software device is clear and not vague along with a temporary design decision at least. SDS can include a reference on the other document like the Software Detailed Design.

(1) Work of verification and Validation

1) Definition

Verification means “confirmation that specific requirements are met by presentation of objective evidence and inspection”. The software verification in the software development environment is confirmation that the output in the specific stage of development meets every input requisite of the said stage. The software test is one of the several verification activities to confirm that the output in the software development stage meets the input requisite. The software validation is to prove with objective evidence that both the user requirements and use purpose on the function whose software has been used are consistent with the software.

Software verification and validation shall be confirmed through the software test. The software test shall conduct the unit test, integration and system test.

Software Configuration Management

Purpose of the configuration management plan

Software should include a function that can materialize the optimum performance on the product developed by this company. To this end, continual upgrade from the previously launched version should be available.

How to manage configuration

1) The development request firm shall conduct monitoring continually the customer requirements, item for performance improvement and other problematic issue, etc on the software-applied product so settle through the software upgrade based on the immediate foregoing.

2) In case the software upgrade have settled the performance improvement, application of the customer request and bug matter, etc, it should be able to distinguish by a version name and write as the following type.

Major : A significant change (for example : the change of PCB or other basic functional frame)

Internal management: Slight change

If Ver. was upgraded for the change of customer requirements and performance improvement, it will be changed accordingly.

 (However, the upgrade item of Ver. will be limited to the one that does not affect the product's basic performance.)

3) In case the software configuration is changed, it should be recorded, managed on 12. Software Configuration Management of the Software Validation Report .

In case the software configuration is changed, it should be validated before it is applied to the software.

Problem settlement and support in the software development activities

Software's problem-solving procedure

The customer dissatisfaction item should be received pursuant to Customer complaints Procedure and if a software change is needed, it should quickly be notified to R&D team.

To settle the software's problem, a top priority should be given to improving the problem under mutual cooperation between Production team and R&D team and it should promptly be applied to the product.

Any issues identified with the software shall be recorded and applicable corrective measures shall be conducted.

In addition, when correcting the issues of software, the company shall examine the issues, notify them to relevant personnel, and analyze any trend of issues. The software shall be used only after the required validation has completed.

As for the software's preservation, R&D team should keep the same version against the original, and whenever the version changes the two firms should put it in a separate backup file (CD, hard disk) so keep in the system.
