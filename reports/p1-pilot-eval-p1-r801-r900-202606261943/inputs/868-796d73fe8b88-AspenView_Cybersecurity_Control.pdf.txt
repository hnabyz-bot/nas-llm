7.3 Category of Mitigation Measure
For risk mitigation measures that have been implemented in 7.2, this section will outline the
different categories of controls that have been put in place as a result of those mitigation
measures.
7.3.1 Authentication Controls
o Authentication controls have been significantly strengthened for AspenView by increasing
the complexity of password requirement, locking users out after repeated trying to login
unsuccessfully, and ensuring that a single account can only have a single session logged
into the software.
7.3.2 Authorization Controls
o Authorization controls have been reinforced to ensure that the software integrity and the
data integrity can be maintained. Updates to the software can only be made by an
administrator, and an admin can only patch the software with a set of files that need to be
validated by a password from the manufacturer.
7.3.3 Cryptography Controls
o Cryptography controls of AspenView have been improved to ensure that the sensitive
data that the program handles are safe from external threats. All patient data is encrypted
using DICOM standards, and credentials are encrypted with SHA-256 method to ensure
decryption cannot happen without manufacturer’s decryption methods.

1. Personnel Responsible
• Security Individual Contractor (Dong Hyun Kim): Overall responsibility for cybersecurity management,
responsible for monitoring, identifying, and addressing vulnerabilities with the help of the development
team.
2. Sources, Methods, and Frequency for Monitoring and Identifying Vulnerabilities
• Sources:
o National Institute of Standards and Technology (NIST) National Vulnerability Database (NVD)
o Cybersecurity and Infrastructure Security Agency (CISA) Known Exploited Vulnerabilities Catalog
o Third-party software manufacturers
o Independent security researchers and external cybersecurity advisories
• Methods:
o Continuous monitoring using automated tools (e.g., vulnerability scanners)
o Regular manual reviews and audits by the Security Contractor
o Collaboration with external security researchers and participation in vulnerability disclosure
programs
• Frequency:
o Monthly checks for updates from NIST NVD and CISA
o Weekly reviews of third-party software updates and advisories
o Quarterly comprehensive security audits and assessments
3. Identify and Address Vulnerabilities in CISA Known Exploited Vulnerabilities Catalog
• Identification:
o Automated tools configured to monitor the CISA Known Exploited Vulnerabilities Catalog.
o Manual checks performed weekly by the cybersecurity team.
• Addressing Vulnerabilities:
o Immediate prioritization and analysis of identified vulnerabilities.
o Development and testing of patches or mitigation strategies.
o Deployment of patches following the established update process.
4. Periodic Security Testing
• Frequency:
o Quarterly vulnerability assessments.
o Semi-annual penetration testing.
o Annual comprehensive security audits.
• Methods:
o Use of automated vulnerability scanning tools.
o Manual penetration testing by certified cybersecurity professionals.
o Collaboration with third-party security firms for independent assessments.
5. Timeline to Develop and Release Patches
• Critical Vulnerabilities: Patches developed and released within 1 week.
• High-Risk Vulnerabilities: Patches developed and released within 2 weeks.

6. Update Processes
• Development:
o Secure coding practices followed during patch development.
o Thorough testing in a staging environment before release.
• Deployment:
o Use of automated deployment tools to ensure timely and consistent patch delivery, which can be
only accepted by administrative users of the program. Manual deployment can also be enforced by
releasing the new version files only to the administrative users who can install.
o Rollback mechanisms in place to revert updates if issues are detected.
• Verification:
o Post-deployment verification to ensure patches are applied correctly and systems remain stable.
7. Patching Capability
• Rate of Delivery:
o Automated patch management system capable of delivering updates to all devices within 24 hours
of patch release.
o Scalability to handle patch deployment across all customer environments efficiently.
8. Coordinated Vulnerability Disclosure and Related Processes
• Vulnerability Reporting:
o Publicly accessible vulnerability disclosure policy on our website.
o Dedicated contact point (email and web form) for receiving vulnerability reports.
• Coordination:
o Collaboration with independent researchers and security organizations.
o Participation in industry forums and information sharing initiatives.
• Response:
o Acknowledgement of reports within 48 hours.
o Initial assessment and triage within 240 hours.
o Regular updates to the reporting party on the status of the investigation and remediation efforts.
9. Communication of Remediations, Patches, and Updates to Customers
• Pre-Release Communication:
o Advance notifications to customers about forthcoming patches and updates, including details on
the nature of the vulnerabilities and expected release dates.
• Release Communication:
o Detailed release notes accompanying patches and updates, explaining the vulnerabilities
addressed and the steps customers need to take.
o Use of multiple communication channels, including email, customer portals, and in-application
notifications.
• Post-Release Support:
o Technical support available to assist with patch implementation and address any issues.
