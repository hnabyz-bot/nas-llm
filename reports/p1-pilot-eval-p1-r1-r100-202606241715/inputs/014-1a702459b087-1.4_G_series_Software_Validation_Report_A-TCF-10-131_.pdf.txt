Software Description
AXD is software of Detector that performs image acquisition, image correction, and pre-processing.
AXD starts by opening Main Window, and it is converted to sub modes by selection. And supporting sub modes are as
below.
 Common functions module: Control detector setup
 Image acquisition module: Control image read
 Calibration module: Calibration setup and pre-pre-processing control
 Firmware update: Write detector firmware update file and read information
 iMX8 Module: System initialization and task creation.
And power device and shock sensor communication control and initialization
 FPGA App Module: Image reading and communication message processing in FPGA

Software
Software Function
 Common functions module
The module communicates with the detector through a network socket communication module. Common functions
module provides commands to control detector hardware such as system initialization, window time setting for image
acquisition and detector operation mode setting, and exposure time setting. It also receives feedback on the condition of
the hardware from the detector.
 Image acquisition module
The Image Acquisition module registers the event handle setting for acquisition of image capture images, data
processing event settings, and notification processing messages for image acquisition, and stores the raw images read
from the detector.
 Calibration module
For proper image correction, information of the beam delivery efficiency of each fiber bundle is very important.
Therefore, information of the beam delivery efficiency of each fiber will be provided in the calibration dataset. The
calibration dataset will produce a table of deviation of fiber bundle beam delivery efficiency, and apply it to acquired
images for flat fielding of the image.
 Firmware update module
The Firmware update module provides a function to read the firmware information of the detector hardware or write the
latest firmware and provides commands to manage the firmware of the detector hardware.
FIRMWARE Function
 iMX8 module
When communicating with the SDK, the iMX8 module creates each task that manages and controls the detector device
and supports reading and writing log data related to the shock sensor. And supports i2c expansion controller control for
peripheral devices, power supply communication control, and shock sensor-related chip initialization and register
read/write functions.
 FPGA App module
The FPGA App module sends and receives messages when communicating between the CPU and the FPGA and
supports functions to transmit image acquisition commands to the FPGA, set the operation mode, and store and process
the transmitted image data.
Main Features
Easy and Rapid Work Process.
 Fast image pre-processing
 Multitasking.
 Easy and fast detector configuration.

Application Algorithm and Technology
Below are algorithm, mathematical expression and technology used in software.
Calibration
Calibration performs following compensations to acquired images.
Offset Correction
Obtain dark image which noise is filtered. And then, subtract each pixel value of dark image from real X-ray acquired
image. This correction process makes dark signal offset uniform, and enables to get uniform image.
Gain Correction
Gain correction is calculated by following the steps below. First, a calibration set is created based on the average value
of the acquired n bright data. The input data is then divided by the generated calibration set data, and the mean value of
the calibration set data is multiplied to create the gain map.
This process reduces brightness variations across the entire image and achieves a uniform background brightness.
Bad-Pixel(Defect) Correction
In case that the scanner can’t detect normal intensity light because of alien object in optical fiber collection part or X-
ray storage phosphor plate, extract corresponding area and compensate through dust remover and defect correction
algorithm.

Software Life Cycle
Software development process at software follows a classic waterfall methodology, as illustrated in the following
chart
Analysis. The activity consists of establishing requirements for as much as possible before the design phase. The requirements are
documented in the software-specific requirements and design notes. When enough of the requirements are gathered and analyzed for
this iteration of the prototype, the activities shift into the design phase.
Planning. This activity constructs the software development plan. This plan outlines the tasks, responsibilities, resources, and other
items pertinent to the specific development project.
Design. This activity considers four functional attributes of the software: data structure, software architecture, functionality, and
interface characterization. The design is documented in design documents.
Implementation. The implementation phase converts the design into a machine readable form. The software source code is created
in accordance with AXD 's coding conventions. To verify that the code created is consistent with Company standards, design
reviews/inspections are held.
Qualification. The qualification phase verifies that the design and code implemented meet the requirements. Testing ensures that
defined input will produce actual results that agree with required results.
Maintenance. Once the software enters a released state, the only software changes allowed are to fix identified defects in the code or
to support approved enhancements. All identified system software defects are recorded on System Discrepancy Reports (SDR) forms
or submitted electronically to the Modification Request (MR) database.

Module Test (Verification)
In principle, each requirement from the specifications as well as from risk analysis must be verified by means of an
appropriate test procedure.
The following methods are used:
 Testing at component and/or system level
 Code inspection (for untestable requirements)
 Reviewing, examination of the documentation
Verification of Components
The development of software components takes place in a cyclical process of implementing and testing of components.
The test is performed by the programmer involved. The basis is the requirements assigned to the components and any
test methods and acceptance criteria additionally specified in the design document.
When all requirements have been met, the components are authorized and/or made available for use in the integration
It is specified for each component in the design document, whether it is to be subjected to a component test or whether
it is to be verified in the system test in interaction with other components. When a component test is performed, test
documentation must be created.
This is the project leader’s responsibility.
System Verification
The verification of the final product is done by means of simulated client. The fulfilling of system requirements is to be
verified by means of one or more test procedures. Documentation must be created.
The proof of a subset of requirements can be done in component tests. In such a case the test results for each component
in question must be submitted.
The authorization / rejection of the software takes place after the test results have been submitted in the verification
review. The development leader, head of sales and quality assurance official or a representative must take part in this.
It is tested whether the software coincides with the demands and whether the validation phase can be proceeded with.
The protocol for the verification is an informal document (minimum contents: participants, date, software version
involved, result of the discussions, new tasks with deadlines and responsibilities, software version).
Test Documentation
The documentation of a test must be done in such a way that the test is made comprehensible and repeatable.
A test procedure is drawn up in preparation of the test. The document must contain a description of how the test has
been drawn up and of the test data, as well as an overview of the test procedure. In addition, the test steps to be
performed and their expected results are listed in tabular form.
During the test, the tester enters the following data into a copy or printout of the test procedure (in handwriting):
 The name of the tester, data, software version, test system

 Performing the test step
