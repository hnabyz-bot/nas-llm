1. Overview
1.1 Purpose
The purpose of this document is to show a cyber security architecture view for Digital X-ray
Imaging System (Model name: HnX-R1).
1.2 Scope
This document is limited to HnX-R1, the Digital X-ray Imaging System.
In addition, three of the four architectural views corresponding to the FDA requirements are Global
system view, Updateability/Patchability view, and Security use case view. Multi patient harm view
is excluded because it is not our product.
2. Architecture view
2.1 Data Flow Diagram (Global security architecture view)
A Data-Flow Diagram (DFD) in medical device cybersecurity is a graphical representation that maps
out the flow of data within a medical system. It highlights how data moves between devices, systems,
and users, identifying points where data is input, processed, stored, and output. By visualizing these
flows, DFDs help identify potential vulnerabilities in data handling, transmission, and storage

2.2 Updateability/Patch-ability View
• The detector's update process can only proceed after a proper connection has been established
through the standard communication procedure. No additional ports are used.
1. The console sends the update file (*.bpd) to the detector and initiates the patch request.
(Console → Communication → Communication Interface → Request → Firmware
process in Detector)
2. The detector decrypts the received patch data and performs a SHA-256 integrity check.
3. If the above process completes successfully, a normal response is sent back; otherwise, an
error response is returned.
(Firmware process in Detector → Response → Communication Interface →
Communication → Console)
4. After sending a normal response, the detector proceeds with applying the update and
automatically reboots itself.
