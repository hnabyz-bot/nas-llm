Document overview:

The difference between the GT1717C and GT1717CP products is the internal CsI deposition thickness, but the software and firmware are the same.

The software of the GT1717C and GT1717CP is structured by removing unnecessary modules from the modularized functions of the G-series software architecture.

Software Description

SDK/On Board Module (firmware of CPU module / FPGA) is a pipeline SW bundle that consist of the feature performing image acquisition, image correction, preprocessing, etc., and consists of functions such as manipulation and transfer.

Detector software can be broadly divided into two categories.

1. SDK (Software Development Kit): A collection of functions that acquire detector images and perform various manipulations. This is ultimately compiled into a dynamic link library (DLL) and includes various functions (image correction, preprocessing) that enable the final display of patient images on a PC using a console or other interface.

2. Onboard Module (Firmware/FPGA): This program, embedded in the detector, communicates with the PC console and acquires images from the detector's internal FPGA connected to the panel, reconstructs them into a two-dimensional pixel array, and transmits them. This software is divided into CPU board firmware and FPGA programming.

The SDK consists of a bundles of functions that are executed when called from the console. And supporting sub modes are as follows.

Common functions module: Control detector setup

Image acquisition module: Control image read

Calibration module: Calibration setup and correction or pre-processing control of image

Firmware update: Write detector firmware update file and read information

The On Boards Module (firmware / FPGA) performs the following roles:

CPU Module: System initialization and task creation.

And power device and shock sensor communication control and initialization

FPGA App Module: Image reading and communication message processing in FPGA

Software

SDK(Software Development Kit for Image Acquisition using detector) Function

 Common functions module

The module communicates with the detector through a network socket communication module. Common functions module provides commands to control detector hardware such as system initialization, window time setting for image acquisition and detector operation mode setting, and exposure time setting. It also receives feedback on the condition of the hardware from the detector.

 Image acquisition module

The Image Acquisition module registers the event handle setting for acquisition of image capture images, data processing event settings, and notification processing messages for image acquisition, and stores the raw images read from the detector.

 Calibration module

For proper image correction, information of the beam delivery efficiency of each fiber bundle is very important. Therefore, information of the beam delivery efficiency of each fiber will be provided in the calibration dataset. The calibration dataset will produce a table of deviation of fiber bundle beam delivery efficiency and apply it to acquired images for flat fielding of the image.

On Boards Module update module

The Firmware update module provides a function to read the firmware information of the detector hardware or write the latest firmware and provides commands to manage the firmware of the detector hardware.

On Board Module Function

 Firmware (CPU module)

When communicating with the SDK, the CPU module creates each task that manages and controls the detector device and supports reading and writing log data related to the shock sensor. And supports i2c expansion controller control for peripheral devices, power supply communication control, and shock sensor-related chip initialization and register read/write functions.

 FPGA module

The FPGA App module sends and receives messages when communicating between the CPU and the FPGA and supports functions to transmit image acquisition commands to the FPGA, set the operation mode, and store and process the transmitted image data.

Application Algorithm and Technology

Below are algorithm, mathematical expression and technology used in software.

Calibration

Calibration performs following compensations to acquired images.

Offset Correction

Obtain dark image which noise is filtered. And then, subtract each pixel value of dark image from real X-ray acquired image. This correction process makes dark signal offset uniform, and enables to get uniform image.

Gain Correction

Gain correction is calculated by following the steps below. First, a calibration set is created based on the average value of the acquired n bright data. The input data is then divided by the generated calibration set data, and the mean value of the calibration set data is multiplied to create the gain map.

This process reduces brightness variations across the entire image and achieves a uniform background brightness.

Defect Correction

In case that the scanner can’t detect normal intensity light because of alien object in optical fiber collection part or X-ray storage phosphor plate, extract corresponding area and compensate through dust remover and defect correction algorithm.

 Architecture Design Chart

SDK/On Boards Module Function Diagram and Description

Description

SDK/On Boards Module Software architecture consists of five independent modules. Acquisition modules provide the ability to correct and process raw images obtained from detectors.

Diagram

Software(SDK) Function

Common functions module

The common functions module communicates with the detector through the network socket communications module. The common functions module provides commands for controlling detector hardware, including system initialization, window time settings for image acquisition, detector operational mode settings, and exposure time settings. Also provide hardware status feedback with the detector.

Image acquisition module

The Image acquisition module registers event handle setting and data processing event setting for acquiring a video image, and notification processing messages about image acquisition, and stores the raw image read from the detector.

Calibration module

For proper image correction, information of the beam delivery efficiency of each fiber bundle is very important. Therefore, information of the beam delivery efficiency of each fiber will be provided in the calibration dataset. The calibration dataset will produce a table of deviation of fiber bundle beam delivery efficiency, and apply it to acquired images for flat fielding of the image.

Firmware update module

The Firmware update module provides the ability to load detector hardware firmware information, use the latest firmware, and provide commands to manage detector hardware firmware.

Firmware(On Boards Module) Function

Firmware of CPU Modules

The iMX8 module supports the ability to manage and control the detector equipment during communication with the SDK and to read and write log data associated with impact sensors, and supports i2c expansion controller control and power supply communication control for peripheral devices, initialization of shock sensor related chips, and register read/write capabilities.

Software Development Environment Description

Introduction

This procedure manual (PM) describes the procedure for the development process. It will allow medical software to meet all legal requirements when being developed. Development faults will be avoided by means of a planned development process with an integral risk management system. A description is given on how the necessary testing, verification and validation steps are to be taken, from the Software Requirement Specification through the release of the software and how changes to the software are handled. There are also guidelines for incorporating amendments. The development process fulfils the requirements 21 CFR 820.30 Design Controls.

Concepts

Verification (suitability for the application)

Verification confirms that the objectively defined requirements have been met.

Validation (suitability for implementation)

This confirms that software has been tested for usability against all items in the requirements specification. It must also satisfy all the internal requirements and demands of the user.

PEMS: Programmable electrical medical system

PESS: Programmable electronic sub-system

Responsibilities

Management

Issuing a development instruction

Approving the planned budgets

Evaluating and authorizing the requirement specifications (development tasks)

Authorizing the project plan and assessing observance

Initiating testing, verification and validation of the development and authorization of individual development outcomes as they occur

Coordination of programming for the individual packages

Obtaining the currently valid norms for projects

Approving amendments

Authorizing risk analysis

Distribution

Obtaining and assessing market information

Performing application tests with potential clients to validate the products

Quality Supervisor

Ensuring that the documentation is complete.

Software Life Cycle

Software development process at software follows a classic waterfall methodology, as illustrated in the following chart

Analysis. The activity consists of establishing requirements for as much as possible before the design phase. The requirements are documented in the software-specific requirements and design notes. When enough of the requirements are gathered and analyzed for this iteration of the prototype, the activities shift into the design phase.

Planning. This activity constructs the software development plan. This plan outlines the tasks, responsibilities, resources, and other items pertinent to the specific development project.

Design. This activity considers four functional attributes of the software: data structure, software architecture, functionality, and interface characterization. The design is documented in design documents.

Implementation. The implementation phase converts the design into a machine readable form. The software source code is created in accordance with AXD's coding conventions. To verify that the code created is consistent with Company standards, design reviews/inspections are held.

Qualification. The qualification phase verifies that the design and code implemented meet the requirements. Testing ensures that defined input will produce actual results that agree with required results.

Maintenance. Once the software enters a released state, the only software changes allowed are to fix identified defects in the code or to support approved enhancements. All identified system software defects are recorded on System Discrepancy Reports (SDR) forms or submitted electronically to the Modification Request (MR) database.

Module Test (Verification)

In principle, each requirement from the specifications as well as from risk analysis must be verified by means of an appropriate test procedure.

The following methods are used:

Testing at component and/or system level

Code inspection (for untestable requirements)

Reviewing, examination of the documentation

Verification of Components

The development of software components takes place in a cyclical process of implementing and testing of components. The test is performed by the programmer involved. The basis is the requirements assigned to the components and any test methods and acceptance criteria additionally specified in the design document.

When all requirements have been met, the components are authorized and/or made available for use in the integration

It is specified for each component in the design document, whether it is to be subjected to a component test or whether it is to be verified in the system test in interaction with other components. When a component test is performed, test documentation must be created.

This is the project leader’s responsibility.

System Verification

The verification of the final product is done by means of simulated client. The fulfilling of system requirements is to be verified by means of one or more test procedures. Documentation must be created.

The proof of a subset of requirements can be done in component tests. In such a case the test results for each component in question must be submitted.

The authorization / rejection of the software takes place after the test results have been submitted in the verification review. The development leader, head of sales and quality assurance official or a representative must take part in this. It is tested whether the software coincides with the demands and whether the validation phase can be proceeded with.

The protocol for the verification is an informal document (minimum contents: participants, date, software version involved, result of the discussions, new tasks with deadlines and responsibilities, software version).

Test Documentation

The documentation of a test must be done in such a way that the test is made comprehensible and repeatable.

A test procedure is drawn up in preparation of the test. The document must contain a description of how the test has been drawn up and of the test data, as well as an overview of the test procedure. In addition, the test steps to be performed and their expected results are listed in tabular form.

During the test, the tester enters the following data into a copy or printout of the test procedure (in handwriting):

The name of the tester, data, software version, test system

Performing the test step

Result of the step ("OK“, if the expected result occurs)

Deviations from the test procedure (e.g. where steps are superfluous or further steps are advisable).

Test data, test software and log files must be archived in a data carrier.

System Test (Validation)

Requirements and testing for validation are laid down as for verification, whereby the validation tests are by definition restricted to the use of a possible end user.

Participants in validation are: developments leader, head of sales, quality assurance official.

For validating the end product a simulated end user test, whereby the intended use is imitated by using simulation software and corresponding hardware and following the instruction manual and service documentation.

Then the conformity assessment is conducted. The validation of the product is repeated with a real user test (by clients) with the provision that the product has not yet been certified. By the systematic application of the medical product observation and reporting system, the result of the validation is continuously reviewed and hardened up.

Particularly relevant observations regarding the product’s behavior under working conditions (e.g. by evaluating the service reports etc.) are regularly evaluated by the development head in collaboration with the security official for medical products. In addition, the risk analysis is assessed and expanded in each case if needed.

Amendments

After completion of the development and authorization of the zero series, the product may basically only be changed or modified by a development change. Changes are hardening the source code as well as new minor and lead to a service pack. Debugging is corrective measure for removing faults and is not defined as a development change.

Development changes only occur when the client’s product requirements or marginal technical conditions change. The requirement specifications for changes are laid down in the minutes of the developer meetings. There, all requirements are specified and tested at the same time for any possible new risks. If so, the risk analysis is adjusted.

The amendments must be documented and are added to the main product file.

new conformity statement (see manual)

new manual

filing the program CD

change files with description of changes to software

Any amendment is concluded when a new service pack is released.

Documentation / Main Product File / Risk Management File

A document contains a revision status / amendment index, date and name of originator as well as the controller; it can only be entered into the EDP as a data record. A document is reviewed at regular intervals for validity.

Remarks / minutes are the results of processes and procedures (output). They need at least the project number, date and name of the originator. Preliminary development results (documents and notes) enter the development file.

On completion of the development, all documents and notes must be available and listed in order to confirm the actual state of the released product. The main product file must be submitted for review by an appointed office. The documents of the main product file need not be tangible; a reference to where the corresponding documents can be found will suffice.

The main product file must be administered by the development head. They must be kept on file for at least five years after the sale of the last product.
