3. 퍼징시험 (Fuzz Testing): 로컬테스트로 웹에 올린 것처럼 해서 진행 내구성 스트레스 테스트 : 무작위의 비정상적인 데이터를 대량 입력하여 예외적 오작동을 탐지 모든 입력창에 수십개의 데이터 입력을 해보고 제대로 작동하는지 확인 미지의 취약점 파악 정확한 경로 ( 어떠한 파일의 어떠한 확장자를 몇 번 업로드 했는지에 대한 기록 ) prod 환경에서 진행해야 하지만 서버를 이용하면 비용에 대한 부담이 크기 때문에 실제 서버에서 돌렸는지 로컬에서 돌렸는지 알 수 없으므로 로컬 서버에서 진행하고 그에 대한 기록과 로그를 남기면 됨 정상적인 열쇠가 아님에도 완전히 고장나지 않는지 확인 Input validation Test: 비정상적인 입력값 처리 안전성 검증 의도적으로 비정상적인 데이터를 입력하여 시스템이 다운되거나 정보를 유출하는 대신 “ 잘못된 입력입니다 .” 라고 안전하게 처리하는지 확인 검증로직 사용 , 서버상에서도 한번 더 검증 클라이언트 + 경계값 분석 소스코드 통일화 약속된 코드형식이 아니거나 타입의 길이가 다르면 차단 API Fuzzing Test: API 엔드포인트 비정상 입력 대응 테스트 소프트웨어 특정 기능이나 어플리케이션을 다른데서 쓸 수 있도록 시스템의 내부 통신 규칙 (API) 에 맞지 않는 엉뚱한 요청을 대량으로 보내 숨겨진 취약점이나 시스템의 오작동을 유발 API 명세서 Memory Leak Test: 시스템 메모리 취약점 및 누수 탐지 소프트웨어가 사용했던 메모리를 제대로 반납하지 않아 , 시간이 지날수록 시스템이 점점 느려지다가 결국 멈추게 되는 현상 테스트 시스템이 사용할 수 있는 메모리가 고갈되어 다운되는 것 방지 위함 전문 메모리 프로파일링 도구를 활용하여 테스트 전후 메모리 사용량 확인 어떤 종류의 비정상적인 요청에 의해 시스템이 다운되었는지 상세히 기록

4. 시스템 레벨 보안 시험 (System-level security testing): 보안 기능이 다른 기능과 충돌이나 부작용을 일으키지 않는지 확인 E2E Test: 엔드투엔드 시나리오 기본 보안 검증 실제 사용자의 전체 시나리오 ( 환자가 병원에 와서 진료받고 수납하고 퇴원하는 전 과정 ) 를 그대로 따라가며 그 과정속에서 발생할 수 있는 보안 허점 탐지 시스템 간의 관리 문제 각 단계별로 기대되는 보안상태에 대해 이를 만족하며 각 단계를 정확히 통과했는지 실제 사용자 연계 테스트 Logging Test: 로깅 시스템 적정성 및 보안성 평가 모든 중요한 활동 ( 누가 언제 로그인했는지 , 어떤 데이터를 수정했는지 등 ) 이 빠짐없이 그리고 위변조 불가능하게 기록되는지를 검증 Performance Test: 부하 상황에서의 보안 매커니즘 안전성 평가 병원에 튼튼한 방화문을 설치한 뒤 평상시에 의사나 환자들이 문을 여닫는데 너무 많은 시간이 걸리지는 않는지 ( 성능 ), 갑자기 수백 명의 인파가 몰려들었을 때 ( DDos ) 건물 전체가 마비되지 않고 응급실 만이라도 운영될 수 있는지 ( 가용성 ) 확인하는 것 시스템 자체에 대기번호 부여 : 성능에 대해 아키텍처로 특정상황에서는 서버에 수용가능한 몇 명만 받고 그 이상은 대기열로 빼는 것
