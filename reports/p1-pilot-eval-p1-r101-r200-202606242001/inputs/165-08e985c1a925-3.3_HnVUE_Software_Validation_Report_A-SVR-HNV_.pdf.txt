3. Software Description
3.1. Outline
3.1.1. Software descriptions
HnVUE, a Digital Flat Panel X-ray Detector (Console Software), is a software used in devices that store,
enlarge, reduce, view, analyze, transmit, and output medical images.
3.1.2. Software layout
3.1.3. Data format
All image data must be transferred and stored as 16-bit little-endian (Intel byte order) two-dimensional arrays
without a specific header. The dimensions of the image must be passed as parameters or values to the
auxiliary procedure in the configuration file. For the stored files, the grayscale range is assumed to be
0..2^16-1 (16-bit non-negative integers). This applies if the software uses signed and unsigned integer types
internally for pixel values. Pixels in raw frames are 16-bit non-negative integers.
3.1.4. Robustness and Performance
HnVUE is developed in C# language, and memory management is handled by the garbage collector of .NET
Framework to prevent memory leaks. The software checks the image storage path and notifies the user that
the specified folder or file should be created or the file should be set if it does not already exist. If new data
cannot be saved to a read-only database file, the property of the database file is changed to writable when
the program starts.

A technique for decomposing an image into multiple resolutions and representing high frequency (details) hierarchically.
Generating Gaussian Pyramid
- Gaussian blur the original image → downsample → create multiple resolutions
Generating Laplacian Pyramid
- Calculate the difference between the Gaussian image at each step and the image up sampled to a higher
resolution
- This difference (high frequency component) becomes the Laplacian image

7. Software Development Environment Description
7.1. Introduction
This procedure manual (PM) describes the procedure for the development process. It will allow medical
software to meet all legal requirements when being developed. Development faults will be avoided by
means of a planned development process with an integral risk management system. A description is given
on how the necessary testing, verification and validation steps are to be taken, from the Software
Requirement Specification through the release of the software and how changes to the software are handled.
There are also guidelines for incorporating amendments. The development process fulfils the requirements
21 CFR 820.30 Design Controls.
7.2. Concepts
Verification (suitability for the application)
ü Verification confirms that the objectively defined requirements have been met.
Validation (suitability for implementation)
ü This confirms that software has been tested for usability against all items in the requirements
specification. It must also satisfy all the internal requirements and demands of the user.
PEMS: Programmable electrical medical system
PESS: Programmable electronic sub-system
7.2.1. Responsibilities
7.2.1.1. Management
ü Issuing a development instruction
ü Approving the planned budgets
ü Evaluating and authorizing the requirement specifications (development tasks)
7.2.1.2. Head of Development
ü Authorizing the project plan and assessing observance
ü Initiating testing, verification and validation of the development and authorization of individual
development outcomes as they occur
ü Coordination of programming for the individual packages

7.3. Software Life Cycle
Software development process at software follows a classic waterfall methodology, as illustrated in the
following chart
Analysis. The activity consists of establishing requirements for as much as possible before the design phase. The
requirements are documented in the software-specific requirements and design notes. When enough of the requirements
are gathered and analyzed for this iteration of the prototype, the activities shift into the design phase.
Planning. This activity constructs the software development plan. This plan outlines the tasks, responsibilities, resources,
and other items pertinent to the specific development project.
Design. This activity considers four functional attributes of the software: data structure, software architecture,
functionality, and interface characterization. The design is documented in design documents.
Implementation. The implementation phase converts the design into a machine readable form. The software source
code is created in accordance with HnVUE's coding conventions. To verify that the code created is consistent with
Company standards, design reviews/inspections are held.
Qualification. The qualification phase verifies that the design and code implemented meet the requirements. Testing
ensures that defined input will produce actual results that agree with required results.
Maintenance. Once the software enters a released state, the only software changes allowed are to fix identified defects
in the code or to support approved enhancements. All identified system software defects are recorded on System
Discrepancy Reports (SDR) forms or submitted electronically to the Modification Request (MR) database.

ASCII Encoding Description
Data bits are a measure of the actual data bits in a transmission. When a computer sends a frame of
information, the actual amount of data may not be the full 8 bits. Standard values for a frame are 5, 7, or 8
bits. Which setting you choose depends on the information you are transmitting. For example, standard
ASCII has values from 0 to 127 (7 bits). Extended ASCII uses 0 to 255 (8 bits). If the data you are
transmitting is standard ASCII, sending 7 bits of data per frame is sufficient for communication. A frame is a
single byte transmission that includes a start/stop bit, data bits, and parity. The actual number of bits
depends on the protocol you choose, so the term "frame" can be used to cover all instances.
